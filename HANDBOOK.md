## 이 핸드북이 답하려는 질문

> "개발 작업의 각 단계를 어떤 모델·어떤 에이전트에 맡기는 것이 가장 가성비가 좋은가? 그리고 그 체계를 한 번 세팅하면 매일 어떻게 운영하는가?"

대답을 한 줄로 줄이면: **마스터 세션은 한 모델로 유지하면서 오케스트레이션만 시키고, 작업 성격에 맞는 모델·역할의 서브에이전트에 위임하라.**

이 한 줄이 왜 옳은지, 어떻게 구체화하는지, 매일 어떻게 굴러가는지가 이 핸드북의 전부다.

## 어떻게 읽을 것인가

이 핸드북은 5부로 구성된다. 처음 두 부(왜·무엇)는 의사결정의 근거이고, 나머지 세 부(어떻게·무엇을 운영·어떻게 진화)는 실행이다. 시간이 없다면 부록 A의 부트스트랩 프롬프트를 Opus 4.7에 붙여넣고 부록 B의 검증 체크리스트로 확인한 뒤, 9장(일상 운영)부터 읽어도 된다.

# 제1부 — 왜 분업인가

## 1장. 단일 모델·단일 세션의 한계

### 1.1 모든 걸 Opus로 돌리면 비싸다

Opus 4.7은 강력하지만 비싸다. 입력 100만 토큰당 $5, 출력 100만 토큰당 $25. 같은 작업을 Sonnet 4.6으로 돌리면 입출력 모두 40% 저렴하고, Haiku 4.5로 돌리면 5배 저렴하다.

문제는 Opus의 강점이 모든 작업에서 의미 있게 발휘되지는 않는다는 점이다. 보일러플레이트 CRUD를 Opus로 짜든 Sonnet으로 짜든 결과는 거의 같다. 코드베이스를 grep하는 작업을 Opus에 맡기는 것은 명백한 낭비다. 작업의 난이도와 모델의 능력 사이에 정합성이 맞아야 비용이 정당화된다.

이 정합성을 사람이 매번 의식하기는 번거롭다. "이건 Sonnet으로 충분한가?"를 매 요청마다 따지는 비용이 실제 모델 비용보다 클 수 있다. 시스템 차원의 자동화가 필요한 이유다.

### 1.2 모든 걸 Haiku로 돌리면 품질이 깨진다

반대 방향으로 가면 한계가 다른 곳에서 나타난다. 아키텍처 결정, 동시성 버그 진단, 보안 리뷰 같은 작업은 깊은 추론이 필요하고, 여기서 모델 능력 차이가 결과 품질에 직격탄으로 영향을 준다. 잘못된 아키텍처 결정 한 번이 두고두고 비용을 만들기 때문에, 비가역적 결정에는 가장 좋은 모델을 써야 한다.

따라서 어느 한 모델로 통일하는 전략은 양쪽 끝에서 손해를 본다. 비싼 모델은 단순 작업에서 과지출, 싼 모델은 복잡 작업에서 품질 손실.

### 1.3 마스터 세션 안에서 `/model` 교체의 숨은 비용

직관적인 해결책은 "한 세션 안에서 작업마다 `/model`로 모델을 바꿔 쓰자"이다. 작동은 한다. 그러나 가성비가 무너진다.

핵심은 **prompt cache가 모델별로 분리된다는 점**이다. Opus 캐시와 Sonnet 캐시는 별개다. 마스터 세션에서 Opus로 작업하다가 Sonnet으로 바꾸면, Sonnet은 그 세션의 누적된 컨텍스트(시스템 프롬프트, CLAUDE.md, 그동안의 대화)를 처음부터 다시 처리해야 한다. 캐시 히트 시 표준 입력가의 10%로 끝났을 처리가 100%로 돌아간다.

여기에 Opus 4.7은 토크나이저까지 새로워서, 같은 텍스트가 1.0~1.35배 토큰을 더 사용한다. Opus 4.7로 교체하면 캐시 미스 + 토큰 인플레이션이 동시에 발생한다.

장시간 마스터 세션에서 모델을 자주 바꾸는 워크플로우는 이 두 가지 비용이 누적되어 빠르게 비싸진다. 개발자의 직관 — "모델을 바꾸면 새로 읽는 비용이 든다" — 은 정확히 옳고, 그 비용은 계량 가능하다.

### 1.4 결론: 단일 세션 안에서의 모델 다양성은 부분 최적해

같은 세션 안에서 모델을 바꾸지 말고, 모델 다양성은 다른 메커니즘으로 표현해야 한다. 그 메커니즘이 서브에이전트다.

## 2장. 마스터-서브에이전트 패러다임

### 2.1 격리된 컨텍스트 윈도우라는 발상

서브에이전트는 마스터와 분리된 컨텍스트 윈도우에서 실행된다. 마스터가 "이 코드베이스에서 X를 찾아라"라고 위임하면, 서브에이전트는 자기 책상에서 도구를 호출하고 결과를 정리해서 요약만 마스터에 돌려준다. 마스터는 그 요약만 받는다.

이 단순한 구조가 두 가지 효과를 낸다.

첫째, **마스터 컨텍스트가 보존된다.** grep 결과 수백 줄, 빌드 로그 전체, 탐색 과정의 시행착오가 마스터에 흘러들지 않는다. 마스터는 명세와 결과만 다루므로 컨텍스트가 안정적이다. 안정적인 컨텍스트는 안정적인 캐시 히트를 의미하고, 이는 매 턴의 입력 비용이 폭주하지 않는다는 뜻이다.

둘째, **각 서브에이전트가 자기 모델·자기 도구로 일한다.** 무거운 추론은 Opus, 표준 구현은 Sonnet, 빠른 탐색은 Haiku. 한 작업 안에서도 단계별로 적절한 모델이 동원된다.

### 2.2 Anthropic 자체 데이터: 분업이 단순히 비용 문제가 아닌 이유

Anthropic의 멀티에이전트 리서치 시스템 사례에서 흥미로운 데이터가 있다. Opus 리드 + Sonnet 워커 구성이 단일 Opus로 같은 작업을 처리할 때보다 복잡 리서치 작업에서 90.2% 더 좋은 성과를 냈다. 단순히 "비용 절감을 위해 약간의 품질을 희생"하는 트레이드오프가 아니라, 분업 자체가 품질 레버였다.

직관적으로 그럴 만하다. 한 모델이 모든 컨텍스트와 모든 결정을 짊어지면 인지 부하가 분산되지 않는다. 조정자가 큰 그림을 잡고, 전문가들이 각자 깊이 있게 자기 영역을 파고드는 구도는 사람 조직에서도 효과적이다. 그 패턴이 LLM 시스템에도 적용된다.

### 2.3 패러다임 요약

마스터 세션의 역할:
- 사용자 의도 해석
- 작업 분해와 라우팅
- 서브에이전트 결과 종합
- 짧은 응답·간단한 결정

서브에이전트의 역할:
- 격리된 책상에서 깊이 있는 단일 영역 작업
- 도구 호출의 결과를 흡수하고 요약만 반환
- 자기 영역의 출력 구조를 일관되게 유지

이 분업이 작동하면 자연스럽게 따라오는 결과가 있다. 마스터는 더 작은 모델로도 충분해진다(라우팅·종합은 그렇게 어려운 일이 아니다). 비싼 모델은 정말 필요한 곳(아키텍처, 동시성 디버깅, 보안 리뷰)에서만 출현한다. 평균 비용은 내려가고 평균 품질은 올라간다.

# 제2부 — 가격과 능력의 지형도

## 3장. 모델별 가격과 특성

### 3.1 현재 세대 모델 (2026년 4월 기준)

| 모델 | 입력 / 출력 ($/MTok) | 주요 특징 |
|---|---|---|
| Opus 4.7 | 5 / 25 | 최강 추론. xhigh effort 지원. 새 토크나이저(1.0~1.35× 토큰). 1M 컨텍스트. |
| Opus 4.6 | 5 / 25 | 직전 플래그십. high effort 기본. 1M 컨텍스트. |
| Sonnet 4.6 | 3 / 15 | Opus 대비 40% 저렴. high effort 기본. 1M 컨텍스트. 대부분 작업의 디폴트. |
| Haiku 4.5 | 1 / 5 | Opus 대비 5× 저렴. 200K 컨텍스트. 대량·단순·고속 작업에 적합. |

출력은 모든 모델에서 입력 가격의 정확히 5배다. 이 비율이 일관되어 예산 추정이 단순해진다.

### 3.2 Opus 4.7의 토크나이저 인플레이션

Opus 4.7은 새 토크나이저를 도입했다. 표시 가격은 4.6과 같지만, 같은 텍스트를 토큰화할 때 1.0~1.35배 더 많은 토큰을 만든다. 평균적으로 코드 작업에서 약 1.08~1.15배, 시스템 프롬프트같이 토크나이저가 까다로워하는 텍스트에서 1.4배 가까이 늘어난 사례도 보고됐다.

실용적 의미: 4.6에서 4.7로 옮길 때 같은 청구서가 나오지 않는다. 표시가는 동일해도 실제 비용은 10~30% 더 나올 수 있다. 마스터 모델 선택에서 이는 무시할 수 없는 변수다.

### 3.3 1M 컨텍스트의 의미

Opus 4.7, Opus 4.6, Sonnet 4.6은 모두 1M 토큰 컨텍스트 윈도우를 표준 가격에 제공한다. 추가 요금 없이 거대 문서·전체 코드베이스를 한 번에 다룰 수 있다.

다만 1M을 채워서 일하는 것이 권장 패턴은 아니다. 컨텍스트가 커질수록 매 턴의 입력 비용이 비례해 커지기 때문이다. 1M은 상한이지 목표가 아니다. 1M까지 쓸 수 있다는 것은 "큰 컨텍스트를 다뤄야 할 때 거부당하지 않는다"는 보장이지, "컨텍스트를 가득 채워라"는 지시가 아니다.

서브에이전트 패턴이 효과적인 이유 중 하나가 여기 있다. 마스터의 컨텍스트를 작게 유지하면 매 턴 비용이 작다. 큰 컨텍스트가 필요한 작업은 서브에이전트가 자기 컨텍스트에서 흡수하고 요약만 돌려준다.

### 3.4 Haiku의 위치

Haiku는 종종 과소평가된다. 200K 컨텍스트는 대부분 일상 작업에 충분하고, 가격은 Opus의 5분의 1이다. 단순 추출, 검색, 분류, 요약 작업에서 Sonnet과 의미 있는 품질 차이를 보이지 않는 경우가 많다.

Haiku의 진짜 가치는 "자주, 많이, 빠르게" 호출되는 작업에 있다. 코드베이스 탐색은 거의 모든 비자명 작업의 첫 단계이므로 호출 빈도가 높다. 이걸 Haiku로 처리하면 Opus 마스터의 부담을 크게 덜 수 있다.

## 4장. effort 레벨의 의미

### 4.1 다섯 단계

Opus 4.7과 Opus 4.6, Sonnet 4.6은 effort 파라미터로 추론 깊이를 조절한다. 사용 가능한 레벨은 모델마다 다르다.

| effort | 의미 | 지원 모델 |
|---|---|---|
| low | 빠르고 저렴, 추론 최소 | Opus 4.7 / 4.6 / Sonnet 4.6 |
| medium | 균형. 표준 코딩에 적합 | Opus 4.7 / 4.6 / Sonnet 4.6 |
| high | 깊은 추론. 복잡 작업의 디폴트 | Opus 4.7 / 4.6 / Sonnet 4.6 |
| xhigh | high와 max 사이. Opus 4.7의 디폴트 | Opus 4.7 전용 |
| max | 최대 추론. 비용·지연 최고 | Opus 4.7 / 4.6 / Sonnet 4.6 |

xhigh가 나오기 전, Opus 사용자는 high와 max 중 하나를 골라야 했다. high는 가끔 부족하고 max는 자주 과하다는 불만이 있어 그 사이가 디폴트로 들어왔다. Claude Code는 Opus 4.7에서 xhigh를 자동 디폴트로 설정한다.

### 4.2 effort는 토큰 예산이 아니라 행동 신호

흔한 오해는 effort를 "토큰 한도"로 보는 것이다. 정확히는 행동 조절 신호다. 같은 effort라도 작업 복잡도에 따라 실제 사용 토큰은 달라진다. effort는 모델에게 "얼마나 깊이 파고들어라"라는 지침을 주는 다이얼이지, "이 토큰 안에서 끝내라"라는 한도가 아니다.

이 차이가 실용적 의미를 가진다. 사소한 작업에 max effort를 걸어도 모델이 폭주하지는 않지만, 사소한 작업에 깊이 들어가려고 하므로 결과가 장황해진다. 반대로 어려운 작업에 low effort를 걸면 표면만 보고 답하는 부실한 결과가 나온다. effort는 작업 복잡도와 정합성 있게 골라야 한다.

### 4.3 모델·effort 조합의 의미 있는 관찰

업계에서 자주 인용되는 관찰: **"low effort Opus 4.7 ≈ medium effort Opus 4.6"**. 같은 결과를 낮은 effort로 얻을 수 있다는 뜻이다. Opus 4.7이 추론 효율이 좋아진 결과인데, 동시에 무조건 xhigh를 박는 것이 낭비라는 신호이기도 하다.

또 하나의 관찰: **컨텍스트 품질이 effort 레벨보다 결과 품질에 더 큰 영향을 준다.** 좋은 컨텍스트를 가진 low effort가 모호한 컨텍스트의 max effort보다 자주 더 나은 결과를 낸다. effort 다이얼을 올리기 전에 컨텍스트를 손보는 것이 우선이다.

### 4.4 Opus 4.7의 적응형 추론

Opus 4.7은 고정 thinking budget 모드가 사라지고 적응형 추론(adaptive thinking)만 지원한다. 모델이 단계마다 추론 필요 여부를 스스로 판단해서, 단순 단계는 빠르게 통과하고 복잡 단계에서만 깊게 사고한다. 사용자는 effort 레벨로 그 적응성의 강도를 조절할 뿐이다.

이것이 의미하는 바: Opus 4.7에 대고 "thinking 끄지 말라"고 명시할 필요가 없다. 알아서 필요한 곳에서 사고한다. 다만 사고 양이 작업마다 달라지므로 토큰 예측이 어려워진다. task budget(베타)이 보조 안전장치로 등장한 이유다.

### 4.5 서브에이전트별 effort 설정의 한계 (2026년 4월 현재)

서브에이전트 frontmatter에서 model 필드는 공식 지원되지만, effort 필드는 모든 환경에서 일관되게 동작하지 않는다. 이는 feature request로 진행 중이며, 일부 비공식 보고에서 "Opus 4.6 한정 effort 필드 지원" 같은 부분 동작이 보고된다.

실용적 대응: 서브에이전트의 추론 깊이는 시스템 프롬프트의 본문 지시("think carefully before responding" / "respond concisely")로 가이드하는 것이 안전하다. 환경 변수 `CLAUDE_CODE_EFFORT_LEVEL`은 세션 전체에 글로벌 적용되므로, 에이전트별 차등 effort에는 적합하지 않다.

## 5장. 개발 단계별 가성비 매트릭스

### 5.1 매트릭스

이 표는 이 핸드북의 핵심 의사결정 도구다. 작업 종류별로 어떤 모델·effort 조합이 가성비 최적인지 정리한 것이다.

| 단계·작업 | 권장 조합 | 이유 |
|---|---|---|
| 마스터 플래닝, 아키텍처 결정 | Opus 4.7 / xhigh (중대 시 max) | 비가역, 후속 비용 레버리지 큼 |
| 이슈·태스크 분해 | Sonnet 4.6 / high | 구조적 사고 필요, 양이 많음 |
| 기술 리서치·스파이크 | Sonnet 4.6 / medium~high | 웹·문서 위주, 깊은 추론 적음 |
| 일반 기능 구현 (스펙 명확) | Sonnet 4.6 / medium | 표준 워크호스 |
| 보일러플레이트, 단순 CRUD | Sonnet 4.6 / low 또는 Haiku 4.5 | 패턴 매칭으로 충분 |
| 단순 버그 수정 (스택트레이스 명확) | Sonnet 4.6 / medium | 컨텍스트가 진단을 결정 |
| 복잡 버그 (race, 메모리, 동시성) | Opus 4.7 / xhigh | 깊은 추론 필요 |
| 단일 파일 리팩토링 | Sonnet 4.6 / medium | 영향 범위 작음 |
| 다중 파일·구조적 리팩토링 | Opus 4.7 / high (어려우면 xhigh) | 영향 범위 추론 필요 |
| 일반 PR 리뷰 | Sonnet 4.6 / high | 빠른 회전, 합리적 정확도 |
| 보안·핵심 PR 리뷰 | Opus 4.7 / xhigh | 놓치면 비용 큰 영역 |
| 단위 테스트 작성 | Sonnet 4.6 / medium | 패턴 따라쓰기 |
| 테스트 전략·E2E 시나리오 | Opus 4.7 / high | 커버리지 사고 필요 |
| 코드 주석, README 업데이트 | Sonnet 4.6 / low 또는 Haiku 4.5 | 단순 글쓰기 |
| API 명세, ADR | Opus 4.7 / high | 후일 참조될 핵심 문서 |
| 코드베이스 탐색·검색 | Haiku 4.5 | 빈도 높음, 단순 작업 |
| 문서·로그 추출·요약 | Haiku 4.5 | 대량·단순 |
| 커밋 메시지·릴리즈 노트 | Haiku 4.5 | 짧은 글, 패턴 명확 |
| 서브에이전트 (검색·추출) | Haiku 4.5 또는 Sonnet 4.6 / low | 격리된 단순 작업 |

### 5.2 매트릭스 해석 원칙

이 표를 외우려 하지 말고 세 가지 원칙으로 기억하라.

**비가역·고난도는 Opus.** 한 번 결정하면 되돌리기 어려운 결정(아키텍처, API 계약), 또는 표면적 추론으로는 해결되지 않는 문제(동시성 버그)는 Opus의 영역이다. 비용이 비싸지만 후속 비용 절감으로 정당화된다.

**일상 실행은 Sonnet.** 명세가 명확한 구현, 표준 리팩토링, 테스트 작성, 일반 PR 리뷰는 Sonnet의 영역이다. 가장 자주 나타나는 작업이고, Sonnet의 가격·품질 균형이 여기서 가장 잘 발휘된다.

**대량·단순·반복은 Haiku.** 탐색, 추출, 요약, 분류, 단순 글쓰기는 Haiku로 충분하다. 5배 저렴하고 충분히 빠르다.

### 5.3 매트릭스를 서브에이전트로 매핑

이 매트릭스를 그대로 8개 서브에이전트로 매핑하면 다음 표가 된다. 6장에서 각 에이전트를 자세히 본다.

| 서브에이전트 | 모델 | 매트릭스 항목 |
|---|---|---|
| architect | Opus 4.7 | 마스터 플래닝, 아키텍처, ADR, API 명세 |
| deep-debugger | Opus 4.7 | 복잡 버그, 동시성 |
| pr-reviewer | Opus 4.7 | 보안·핵심 PR 리뷰 |
| implementer | Sonnet 4.6 | 일반 기능 구현, 단순 버그 수정 |
| refactorer | Sonnet 4.6 | 다중 파일·구조적 리팩토링 |
| test-writer | Sonnet 4.6 | 단위·통합 테스트 |
| doc-writer | Sonnet 4.6 | API 명세, README, 주석, ADR (architect의 결정을 문서화) |
| explorer | Haiku 4.5 | 코드베이스 탐색, 검색 |

이슈·태스크 분해는 별도 에이전트로 두지 않고 마스터가 처리한다(라우팅·계획 수립이 마스터의 본업이므로). 보일러플레이트는 implementer가 low effort 분위기로 처리한다. 일반 PR 리뷰는 마스터가 가볍게 본다.

# 제3부 — 8개 서브에이전트 표준 구성

## 6장. 에이전트 청사진

### 6.1 왜 8개인가

5장의 매트릭스를 자연스럽게 매핑하면 정확히 8개의 에이전트가 나온다. 더 적게 잡으면 역할 경계가 모호해지고, 더 많이 잡으면 라우팅이 어려워진다. 8개는 시작점으로 적당한 크기이고, 도메인 특화가 필요해지면 14장의 가이드를 따라 추가하면 된다.

8개 에이전트는 두 차원으로 나뉜다.

**모델 차원**: Opus 3개 (architect, deep-debugger, pr-reviewer), Sonnet 4개 (implementer, refactorer, test-writer, doc-writer), Haiku 1개 (explorer).

**행위 차원**: 분석·설계만 하는 읽기 전용 에이전트 (architect, pr-reviewer, explorer), 코드를 수정하는 쓰기 가능 에이전트 (deep-debugger, implementer, refactorer, test-writer, doc-writer).

이 두 차원의 조합이 각 에이전트의 정체성을 결정한다.

### 6.2 architect — 비가역적 결정의 깊은 추론자

**모델**: Opus 4.7
**도구**: Read, Glob, Grep, WebSearch, WebFetch (수정 도구 없음)
**용도**: 아키텍처 결정, ADR, API/스키마 설계, 트레이드오프 분석.

architect는 비가역적 결정에 대한 깊은 추론을 전담한다. "이 인증 흐름을 OAuth로 갈지 자체 토큰으로 갈지" 같은 질문을 받으면 옵션을 나열하고 각 옵션의 장단점, 위험, 전제를 정리한 ADR을 산출한다.

핵심 제약: **architect는 코드를 수정하지 않는다.** Edit/Write 도구를 갖지 않는다. 분석과 문서화만 한다. 이 제약이 architect를 진짜 사고하는 도구로 유지시킨다. 만약 architect가 직접 코드를 수정할 수 있다면, "ADR을 쓰는 척하면서 구현해버리는" 사고 단축이 일어나기 쉽다.

출력 구조는 일관되어야 한다: ① 의사결정 요약 ② 옵션 비교 (최소 2개) ③ 선택 근거 ④ 위험·전제 ⑤ 후속 액션. 이 구조가 architect의 결과를 다른 에이전트(특히 implementer)가 받아쓰기 좋게 만든다.

### 6.3 deep-debugger — 단순 스택트레이스를 넘어선 버그 사냥꾼

**모델**: Opus 4.7
**도구**: Read, Grep, Glob, Bash, Edit
**용도**: 동시성, 메모리, 미스터리 버그.

deep-debugger의 description에는 "NOT for simple stack-trace debugging"이 명시된다. 스택트레이스가 명확한 버그는 implementer가 처리한다. deep-debugger는 그 이상 — race condition, 메모리 누수, 환경에서만 재현되는 heisenbug, 모듈 간 상호작용에서 발생하는 미묘한 버그 — 를 받는다.

워크플로우는 가설 기반이다: 가설 수립 → 증거 수집 → 가설 검증 → 최소 수정. 무한 추론 폭주를 막기 위해 시스템 프롬프트에 "가설 3회 연속 실패 시 보고하고 중단"을 박아둔다. 더 어려운 문제로 판단되면 architect로 에스컬레이션하거나 사람의 추가 정보를 받는다.

deep-debugger는 코드 수정 권한을 가진다 (Edit 도구 포함). architect와 달리 진단과 수정을 한 흐름에서 처리하는 것이 효율적이기 때문이다.

### 6.4 pr-reviewer — 놓치면 비싼 영역의 감시자

**모델**: Opus 4.7
**도구**: Read, Grep, Glob, Bash (수정 도구 없음)
**용도**: 보안·핵심·아키텍처 영향 PR 리뷰.

pr-reviewer는 모든 PR을 보지 않는다. 스타일·린트·포맷팅은 도구 영역이고, 일반 기능 PR은 마스터가 가볍게 본다. pr-reviewer는 놓치면 비용이 큰 영역 — 보안, 동시성, 데이터 정합성, 인터페이스 계약, 권한 — 에 집중한다.

출력 구조는 우선순위 기반이다: ① 차단(blocker) 이슈 ② 강력 권고(should-fix) ③ 제안(nice-to-have) ④ 칭찬할 점. 이 구조는 PR 작성자가 무엇부터 고쳐야 할지 즉시 알게 한다.

pr-reviewer도 architect와 마찬가지로 읽기 전용이다. 리뷰어가 직접 고치는 것은 코드 오너십을 모호하게 만들고, 리뷰의 신뢰성도 떨어뜨린다.

### 6.5 implementer — 표준 작업의 디폴트 워크호스

**모델**: Sonnet 4.6
**도구**: Read, Edit, Write, Glob, Grep, Bash
**용도**: 명세가 명확한 기능 구현, 단순 버그 수정.

implementer는 가장 자주 호출되는 에이전트다. 일상 개발 작업의 70% 이상이 여기로 간다. 명세가 명확하면 그대로 구현하고, 명세가 모호하면 작업을 시작하기 전에 마스터에 질의하도록 시스템 프롬프트에 박는다. 이 한 줄이 "implementer가 추측으로 잘못 구현하는" 가장 흔한 실패를 차단한다.

implementer는 시작 시 explorer로 프로젝트의 코딩 관행을 파악하는 패턴을 권장한다. 새 코드는 기존 코드와 동질감을 가져야 하고, 그러려면 기존 패턴을 알아야 한다.

출력 구조: ① 변경 파일 목록 ② 핵심 변경 요약(3줄 이내) ③ 검증 방법. 짧고 일관된 형식이 마스터의 종합 비용을 낮춘다.

### 6.6 refactorer — 동작 보존 변경의 전문가

**모델**: Sonnet 4.6
**도구**: Read, Edit, Write, Glob, Grep, Bash
**용도**: 다중 파일 리팩토링, 구조 개선.

refactorer의 핵심 원칙은 "동작 보존"이다. 리팩토링이 새 기능을 추가하거나 동작을 바꾼다면, 그것은 refactoring이 아니라 일반 변경이고 implementer의 일이다.

refactorer는 한 번에 한 가지 패턴만 적용한다. "이름 바꾸기", "함수 추출", "조건문 다형성으로 치환" 같이 명확한 단일 패턴 단위로 변경하고, 매 단계마다 빌드·테스트가 통과하는지 확인한다. 광범위한 영향이 예상되는 구조 변경은 refactorer 단독으로 시작하지 않고 architect로 에스컬레이션해 ADR을 먼저 쓰는 것이 안전하다.

### 6.7 test-writer — 기존 패턴을 따르는 테스트 작성자

**모델**: Sonnet 4.6
**도구**: Read, Edit, Write, Glob, Grep, Bash
**용도**: 단위, 통합, E2E 테스트 작성.

test-writer는 시작 시 explorer를 활용해 프로젝트의 기존 테스트 디렉토리·관행을 파악한다. 테스트 프레임워크, 명명 규칙, mocking 스타일, fixture 위치 — 이 모든 것이 프로젝트마다 다르다. test-writer가 새 테스트를 만들 때 기존 패턴을 따라야 일관성이 유지된다.

출력 구조: ① 추가된 테스트 파일 ② 커버하는 케이스 요약 ③ 발견한 엣지 케이스. 마지막 항목이 특히 가치 있다 — 테스트를 쓰는 과정에서 미처 생각지 못한 케이스가 드러나는 일이 많기 때문이다.

### 6.8 doc-writer — 사람이 읽을 글의 작성자

**모델**: Sonnet 4.6
**도구**: Read, Edit, Write, Glob, Grep
**용도**: README, API 문서, ADR, 주석.

doc-writer는 사람이 읽을 글을 쓴다. 핵심 원칙: **코드가 진실의 원천**. 추측으로 문서를 쓰지 않고 코드를 먼저 읽는다. 톤과 구조는 프로젝트의 기존 문서를 따른다.

doc-writer는 architect의 결정을 문서화하는 역할도 자주 맡는다. architect가 ADR을 쓰면, 그것이 후일 README나 개발자 가이드로 확장될 때 doc-writer가 받는다. 이 분업이 architect를 "결정하는 사람", doc-writer를 "전달하는 사람"으로 명확히 구분한다.

### 6.9 explorer — 거의 모든 작업의 첫 호출

**모델**: Haiku 4.5
**도구**: Read, Glob, Grep, Bash (수정 도구 없음)
**용도**: 코드베이스 탐색, 파일 발견, 패턴 검색.

explorer는 가장 자주 호출되는 에이전트일 가능성이 높다. 비자명한 작업은 거의 모두 "어디에 무엇이 있는지" 파악으로 시작하기 때문이다. Haiku로 충분하므로 비용은 미미하고, 결과는 다음 단계 에이전트가 받는다.

explorer의 절대 원칙: **코드 수정 금지** (읽기 전용). 그리고 **요약만 반환**. explorer가 원시 grep 결과 수백 줄을 그대로 토해내면 마스터 컨텍스트가 부풀어 서브에이전트 패턴의 의미가 없어진다. 시스템 프롬프트에 출력 구조 — ① 디렉토리 레이아웃 ② 빌드·테스트 명령 ③ 관찰된 관행 ④ 관련 파일 경로 — 를 강하게 박아 압축된 결과만 받는다.

### 6.10 한 페이지 요약

| 에이전트 | 모델 | 쓰기 권한 | 핵심 출력 |
|---|---|---|---|
| architect | Opus 4.7 | × | ADR (의사결정 요약) |
| deep-debugger | Opus 4.7 | ✓ | 근본 원인 + 패치 |
| pr-reviewer | Opus 4.7 | × | 우선순위별 리뷰 |
| implementer | Sonnet 4.6 | ✓ | 변경 파일 + 요약 |
| refactorer | Sonnet 4.6 | ✓ | 적용 패턴 + 변경 |
| test-writer | Sonnet 4.6 | ✓ | 테스트 + 엣지 케이스 |
| doc-writer | Sonnet 4.6 | ✓ | 문서 + 근거 코드 |
| explorer | Haiku 4.5 | × | 압축된 코드베이스 지도 |

## 7장. 라우팅 스킬

### 7.1 스킬과 서브에이전트의 관계

스킬과 서브에이전트는 다른 도구다.

- **서브에이전트** (`.claude/agents/<name>.md`): 위임의 대상. 격리된 컨텍스트에서 실행되는 일꾼.
- **스킬** (`.claude/skills/<name>/SKILL.md`): 마스터의 의사결정을 돕는 절차·지식 묶음. 자체 실행 단위가 아님.

라우팅 스킬은 마스터에게 "언제 어떤 서브에이전트를 호출할지"를 알려주는 역할을 한다. 마스터가 사용자 요청을 받았을 때, 스킬의 결정 트리를 적용해 직접 처리할지·위임할지·누구에게 위임할지를 일관되게 결정하게 한다.

### 7.2 라우팅 스킬의 본문 구조

라우팅 스킬은 다음 요소를 포함한다.

**위임 결정 트리.** 단순 1-2턴 응답이면 마스터 직접 처리. 다단계·도구 호출 多·결과 격리 필요 시 위임. 격리 필요성은 "결과가 길고 마스터 컨텍스트를 더럽힐 수 있는가"로 판단.

**작업 유형 → 에이전트 매핑.** 6장의 8개 에이전트 표가 핵심. 한국어·영어 키워드를 모두 수록해 자연어 요청에서 매핑이 일관되게 일어나도록 한다.

**비용 가이드.** `inherit` 기본값의 위험성, `model` 명시의 중요성. 새 에이전트를 추가할 때마다 사용자가 점검하도록 알림.

**위임 명세 4요소 템플릿.** 의도 / 제약 / 완료 기준 / 관련 파일. 이 4요소가 빠지면 결과 품질이 즉각 떨어진다는 점을 강조.

**라우팅 실패 시그널.** 잘못된 에이전트로 갔을 때의 신호와 회수 방법. 자세한 진단은 13장 참조.

### 7.3 스킬 활성화 동작

스킬은 description 매칭으로 자동 활성화된다. 라우팅 스킬의 description을 "Use to decide whether to delegate a development task..."처럼 일반적인 트리거로 작성하면, 비자명한 개발 요청 대부분에서 자동으로 로드된다.

명시적으로 적용하고 싶으면 사용자가 "라우팅 스킬을 적용해 결정해라" 한 줄을 덧붙이면 된다. 자동 활성화가 일관되지 않을 때 사용하는 강제 트리거다.

# 제4부 — 실행

## 8장. 시스템 부트스트랩

### 8.1 부트스트랩 한 번, 운영 매일

서브에이전트 시스템은 프로젝트 초기에 한 번 셋업하면 이후 매일 활용한다. 셋업의 목표는 다음 산출물을 정확히 만드는 것이다.

```
.claude/
├── agents/
│   ├── architect.md
│   ├── deep-debugger.md
│   ├── pr-reviewer.md
│   ├── implementer.md
│   ├── refactorer.md
│   ├── test-writer.md
│   ├── doc-writer.md
│   └── explorer.md
└── skills/
    └── sub-agent-routing/
        └── SKILL.md

CLAUDE.md (오케스트레이션 가이드 포함)
```

이 셋업을 사람이 손으로 하지 않고 Opus 4.7 자체에 위임한다. 부트스트랩 프롬프트(부록 A)를 Opus 4.7 / xhigh 세션의 첫 메시지로 붙여넣으면 된다.

### 8.2 왜 Opus 4.7 / xhigh로 부트스트랩하는가

부트스트랩은 일회성 작업이지만 후속 모든 개발 효율의 기반이 된다. 한 번 잘못 만들어두면 매 작업마다 비용을 만든다. 충분한 추론과 자체 점검이 필요하므로 가장 좋은 모델·가장 깊은 effort를 쓴다.

부트스트랩 프롬프트는 Opus 4.7의 "literal" 성향에 맞춰 모든 명세를 첫 턴에 명시한다. 의도, 제약, 완료 기준, 파일 경로, 각 에이전트의 frontmatter, 출력 구조 요구사항, 자체 점검 체크리스트까지. 이 명시성이 Opus 4.7에서 모호한 결과를 막는다.

### 8.3 부트스트랩 후 검증

부트스트랩이 끝나면 다음을 확인한다 (부록 B의 체크리스트 참조).

1. `/agents` 명령으로 8개 에이전트가 모두 보이는가.
2. 각 에이전트의 model 필드가 inherit이 아닌 명시적 값인가.
3. explorer로 간단 호출 1회가 출력 구조에 맞게 결과를 내는가.
4. CLAUDE.md에 위임 4요소 템플릿과 한국어 키워드 매핑이 포함되었는가.
5. 라우팅 스킬의 description이 자동 활성화 가능한 트리거를 가지는가.

이 5개가 통과하면 시스템이 살아있다.

### 8.4 기존 프로젝트에 적용할 때

`.claude/` 디렉토리가 이미 있는 프로젝트에 적용할 때는 충돌 점검이 먼저다. 부트스트랩 프롬프트는 이를 명시적으로 지시한다 — Opus 4.7이 explorer로 기존 구조를 파악하고 충돌이 있으면 사용자 승인을 대기한다. 같은 이름의 에이전트가 이미 있다면 덮어쓸지·이름을 바꿀지 결정해야 한다.

CLAUDE.md는 보존이 원칙이다. 프로젝트 고유의 가이드가 이미 있을 가능성이 높으므로, 부트스트랩 프롬프트는 별도 섹션 ("## Sub-agent Orchestration")으로만 추가한다.

## 9장. 일상 운영

### 9.1 호출 방법 세 가지

**자동 라우팅** — 가장 자연스러운 방식. 사용자가 "이 코드베이스에서 X를 찾아줘"라고 말하면 마스터가 description의 키워드를 매칭해 explorer를 호출한다. 매핑이 명확한 작업에서 잘 작동한다.

**명시적 호출** — `@architect 이 설계 검토해줘` 형식. 자동 매칭이 신뢰가 안 가거나 특정 에이전트를 강제하고 싶을 때. 학습 초기에 자주 쓰면 라우팅 패턴을 빠르게 체득할 수 있다.

**라우팅 스킬 강제** — "라우팅 스킬을 적용해 결정해라" 한 줄. 마스터가 직접 처리하려는 경향을 보일 때, 또는 위임 결정에 일관성을 원할 때.

### 9.2 마스터가 직접 처리할 것 vs 위임할 것

직접 처리:
- 한두 줄로 답할 수 있는 질문
- 빠른 사실 확인
- 결과를 즉시 종합해야 하는 짧은 흐름
- 위임 대상이 마땅치 않은 모호한 작업

위임:
- 대규모 코드 탐색
- 다단계 도구 호출이 필요한 작업
- 결과가 길고 마스터 컨텍스트를 더럽힐 수 있는 작업
- Opus급 깊은 추론이 필요한 단일 분야 작업
- 같은 패턴을 반복할 가능성이 있는 작업

마지막 항목이 미묘하다. 같은 종류 작업을 반복할 거라면 첫 번째부터 위임 패턴으로 가는 것이 학습 비용을 한 번에 끝낸다.

### 9.3 위임 명세 4요소 템플릿

서브에이전트에 던지는 메시지는 4요소를 모두 포함해야 한다. Opus 4.7은 명령을 문자 그대로 따르므로 명세가 모호하면 결과도 좁아진다.

```
의도: [무엇을 왜]
제약: [지켜야 할 것 / 건드리면 안 될 것]
완료 기준: [무엇이 되면 끝인가]
관련 파일: [경로 목록 또는 발견 방법]
```

기술 스택을 모르는 상태로 위임할 때는 "관련 파일" 줄에 "explorer로 먼저 확인"을 넣어두면 된다. 위임받은 에이전트가 알아서 탐색 후 진행한다.

### 9.4 결과 받아 종합하기

서브에이전트가 결과를 돌려주면 마스터는 그것을 사용자에게 그대로 전달하지 않는다. 마스터는 두 가지를 한다.

첫째, 사용자 요청과 결과를 매핑해서 빠진 부분이 있는지 본다. 명세가 일부만 만족됐다면 추가 위임을 한다.

둘째, 결과를 사용자 친화적 형태로 종합한다. 서브에이전트의 출력 구조는 다음 단계 에이전트가 받기 좋게 설계되어 있지, 사용자가 읽기에 최적은 아니다. 마스터가 이를 사람이 읽기 좋은 요약·다음 액션 제안으로 다듬는다.

이 종합이 마스터의 본업이다. 잘 분업된 시스템에서 마스터는 코드를 거의 짜지 않고, 종합과 라우팅에 시간을 쓴다.

## 10장. 표준 워크플로우 4종

### 10.1 신규 기능 개발

1. (마스터) 요구사항 정리, 모호점 사용자에게 질의
2. (architect) API/스키마 설계 → ADR 초안
3. (마스터) ADR 검토·승인
4. (explorer) 영향 받는 코드 위치 파악
5. (implementer) 구현
6. (test-writer) 테스트 추가
7. (pr-reviewer) 보안·핵심 검토 (해당 시에만)
8. (doc-writer) README/CHANGELOG 갱신
9. (마스터) 종합 보고

이 흐름의 핵심은 architect의 ADR이 implementer의 입력이 되는 부분이다. ADR이 명확하면 implementer는 거의 자율적으로 구현하고, 사용자는 ADR 단계에서 한 번만 의사결정에 개입하면 된다.

### 10.2 단순 버그 수정

1. (마스터) 스택트레이스 검토
2. (explorer) 관련 코드 위치
3. (implementer) 수정
4. (test-writer) 회귀 테스트

deep-debugger를 부르지 않는다. 스택트레이스가 명확한 버그는 implementer가 처리할 수 있는 일이다. deep-debugger는 비싸고 느리므로 정말 필요할 때만 쓴다.

### 10.3 난해 버그

1. (마스터) 증상 정리, 재현 절차 확보
2. (deep-debugger) 가설 → 증거 → 검증 → 최소 수정
3. (test-writer) 회귀 테스트
4. (마스터) 사후 분석 메모 작성 권유

deep-debugger가 가설 3회 실패로 보고하고 멈추면 architect로 에스컬레이션하거나 사람의 추가 정보를 받는다. 무한 폭주를 막는 안전장치다.

### 10.4 대형 리팩토링

1. (architect) 리팩토링 전략 ADR
2. (explorer) 영향 범위 파악
3. (refactorer) 단계별 리팩토링 (한 번에 한 패턴)
4. (test-writer) 보강 테스트
5. (pr-reviewer) 종합 리뷰

architect가 ADR로 전략을 세우고, refactorer가 그 전략을 구현하는 구도다. refactorer 혼자 시작하면 광범위한 영향을 추론하지 못해 중간에 막힐 수 있다.

### 10.5 워크플로우는 절대 규칙이 아니다

위 4종은 자주 만나는 패턴의 예시다. 실제 작업은 변형이 많다. 핵심 원칙만 기억하면 된다 — **비가역적 결정은 Opus, 표준 작업은 Sonnet, 탐색은 Haiku, 결정은 ADR로 명시한 후 구현으로 넘어간다.**

# 제5부 — 운영과 진화

## 11장. 비용 레버 다섯 가지

### 11.1 명시적 model 지정

frontmatter의 `model` 필드 기본값은 `inherit`(마스터 모델 상속)이다. 이것이 가장 비싼 기본값이다. 마스터가 Opus 4.7이고 4개 에이전트가 inherit이면 모두 Opus 4.7로 돌아간다. 비용은 4배.

해결: 모든 에이전트의 model 필드를 명시적으로 지정한다. 부트스트랩 프롬프트는 이를 강제하지만, 새 에이전트를 추가할 때마다 사람이 다시 점검해야 한다.

### 11.2 Haiku 우선 사용

explorer가 거의 모든 비자명 작업의 첫 호출이라는 점을 다시 강조한다. 탐색을 Haiku로 압축한 결과만 다음 단계(Sonnet/Opus)에 넘기는 것이 표준 패턴이다. 이 한 단계가 평균 비용을 크게 낮춘다.

같은 원리로, 단순 추출·요약·분류는 Haiku 에이전트로 분리하는 것을 검토한다. doc-writer가 모든 문서 작업을 받기보다, "문서·로그에서 특정 정보 추출"은 Haiku 에이전트로 보내는 식.

### 11.3 결과 요약 강제

서브에이전트의 시스템 프롬프트에 "출력 구조"를 박아 결과 길이를 제한한다. 부트스트랩 프롬프트는 모든 에이전트에 출력 구조를 명시하지만, 사용 중 결과가 장황해지면 시스템 프롬프트를 더 강하게 좁혀야 한다.

장황한 결과는 두 가지 비용을 만든다. 직접 비용 — 결과 토큰 자체. 간접 비용 — 마스터 컨텍스트 부풀림. 두 번째가 누적되면 매 턴 입력 비용이 비례해 증가한다.

### 11.4 프롬프트 캐싱 활용

CLAUDE.md, 시스템 프롬프트, 자주 참조하는 파일은 캐시 친화적으로 작성한다. 캐시 히트는 표준 입력가의 10%다. 안정화된 후에는 시스템 프롬프트를 자주 수정하지 않는 것이 핵심 — 수정할 때마다 캐시가 무효화된다.

마스터에서 `/model` 교체 금지 원칙(1장)도 본질적으로 캐시 보호 원칙이다. 모델 교체 = 캐시 분리 = 캐시 무효화.

### 11.5 배치 API

야간 대규모 리팩토링, 대량 문서화, 대량 테스트 생성은 배치 API로 분리하면 추가 50% 절감이다 (응답 24시간 이내 보장). 인터랙티브 세션과 배치 작업의 경계를 의식적으로 설계하라.

배치는 인터랙티브 사용에 적합하지 않지만, "주말에 돌려놓고 월요일 아침에 결과 보기" 식의 작업에는 강력하다. Opus 4.7 배치는 $2.50 / $12.50로 떨어져 Sonnet 4.6 표준가보다 저렴해진다.

## 12장. 자주 빠지는 함정

### 12.1 마스터에서 `/model` 교체

캐시 무효화로 비용 폭증. 절대 하지 말 것. 다른 모델이 필요하면 서브에이전트로 위임한다. 새 마스터 세션을 시작해야 한다면 이전 세션을 정리하고 새로 시작하라.

### 12.2 inherit 방치

새 에이전트를 만들 때 model 필드를 빼먹으면 inherit이 기본값. 마스터가 Opus면 새 에이전트도 Opus가 된다. 항상 명시할 것.

### 12.3 모호한 위임 명세

Opus 4.7은 문자 그대로 따르므로 모호하면 좁은 결과를 낸다. Sonnet도 명세 품질에 비례해 결과 품질이 결정된다. 4요소 템플릿(의도/제약/완료 기준/파일)을 항상 지키라.

### 12.4 architect에게 코드 수정 시키기

architect는 분석·문서화 전용. 수정은 implementer 또는 refactorer. architect의 도구 목록에 Edit/Write가 없는 이유다. 만약 architect가 직접 수정할 수 있게 만들면, "ADR을 쓰는 척하면서 구현해버리는" 사고 단축이 일어나 ADR의 의미가 희석된다.

### 12.5 explorer 결과를 마스터가 다 흡수

explorer가 원시 grep 출력을 토해내면 컨텍스트 부풀림 회피의 의미가 없다. 부트스트랩 프롬프트는 출력 구조를 강제하지만, 사용 중 늘어지면 시스템 프롬프트를 더 좁혀야 한다.

### 12.6 deep-debugger 무한 폭주

가설 3회 실패 시 보고하도록 박아둔다. 그 시점에서 architect로 에스컬레이션하거나 사람의 추가 정보를 받는다. 이 안전장치가 없으면 deep-debugger가 끝없이 가설을 변형하며 토큰을 소모할 수 있다.

### 12.7 모든 PR을 pr-reviewer로 보내기

스타일·린트는 도구 영역. pr-reviewer는 보안·핵심·아키텍처에 한정. 일반 PR은 마스터가 가볍게 본다. pr-reviewer를 모든 PR에 쓰면 비용이 폭증할 뿐 아니라, "정말 중요한 PR"의 신호가 묻힌다.

### 12.8 서브에이전트끼리 협업 기대

서브에이전트는 자식 서브에이전트를 만들 수 없다. 협업 흐름은 항상 마스터가 지휘한다. "explorer로 찾고 implementer로 수정해라"는 마스터의 명령이지, explorer가 implementer를 부르는 게 아니다. 이 제약은 무한 중첩을 막는 안전장치이기도 하다.

## 13장. 정상·이상 신호

### 13.1 정상 신호

- 사용자 요청에 마스터가 짧게 답하고, 무거운 작업은 서브에이전트에 자동 위임된다.
- 서브에이전트 결과가 명세된 출력 구조를 따른다.
- 마스터의 컨텍스트가 며칠 사용해도 안정적으로 유지된다.
- 비용이 작업 난이도에 비례해 예측 가능하다.
- Opus 호출이 전체 호출의 소수에 그친다 (10~20% 정도가 건강한 비율).

### 13.2 이상 신호와 대응

**마스터가 모든 걸 직접 처리한다.** CLAUDE.md의 라우팅 규칙이 약하거나 description이 부정확. 라우팅 스킬을 명시적으로 호출해 본다. 그래도 안 되면 description의 트리거 키워드를 사용 빈도 높은 한국어·영어 표현으로 보강한다.

**잘못된 에이전트로 자주 간다.** description의 트리거가 충돌. 가장 일반적인 에이전트(implementer, explorer)의 description을 다른 에이전트와 구분되도록 좁힌다. "NOT for ..."로 부정 트리거를 추가하는 것이 효과적.

**결과가 장황하다.** 시스템 프롬프트의 출력 구조 지시를 더 강하게 명시한다. "3줄 이내로 요약하라"처럼 구체적 길이 제약을 박는 것도 한 방법.

**비용이 예상보다 높다.** `/agents` 출력에서 inherit이 남아있는지 확인. 마스터 모델을 한 단계 낮추는 것도 고려 (Opus 마스터 → Sonnet 마스터). Opus 호출 비율을 점검하라 — 30% 이상이면 위임이 잘못 라우팅되고 있을 가능성.

**Opus 에이전트가 사소한 작업까지 받는다.** description에 "NOT for ..." 부정 트리거를 명시. 예: pr-reviewer description 끝에 "NOT for style/lint reviews — those belong to tooling"을 추가.

**서브에이전트가 마스터에 자주 질의한다.** 위임 명세 4요소가 빠지고 있을 가능성. 마스터의 위임 패턴을 점검하고 CLAUDE.md의 템플릿을 다시 학습하도록 한다.

## 14장. 시스템 진화시키기

### 14.1 8개 구성은 출발점이다

이 핸드북의 8개 에이전트 구성은 범용적인 시작점이다. 종착점이 아니다. 프로젝트 도메인이 명확해지면 다음 방향으로 진화시킨다.

### 14.2 도메인 특화 에이전트 추가

자주 만나는 도메인 작업이 있다면 전용 에이전트로 분리한다.

- **db-migrator** (Sonnet 또는 Opus): DB 마이그레이션 전문. 스키마 변경, 데이터 백필.
- **security-auditor** (Opus): 보안 감사 전문. 취약점 스캔 결과 해석, 권한 모델 점검.
- **perf-profiler** (Opus): 성능 분석. 프로파일링 결과 해석, 핫스팟 식별.
- **api-designer** (Opus): REST/GraphQL/gRPC 등 API 설계 전문. architect의 하위 분화.
- **release-manager** (Sonnet): 릴리즈 준비 자동화.

새 에이전트를 만들 때마다 model 필드 명시를 잊지 말 것 (12.2 함정).

### 14.3 외부 도구 연결 (MCP)

특정 서브에이전트에 MCP 서버를 마운트하면 그 에이전트만 외부 도구에 접근할 수 있다. 보안과 비용 양면에서 격리가 강화된다.

예: db-migrator에만 데이터베이스 MCP 서버를 마운트. 다른 에이전트는 DB에 접근할 수 없으므로 실수로 프로덕션 DB를 건드리는 위험이 차단된다.

### 14.4 워크플로우를 SKILL로 추출

자주 반복되는 워크플로우를 SKILL로 추출하면 매번 마스터가 같은 결정을 반복하지 않아도 된다.

예: "릴리즈 준비" 스킬이 ① CHANGELOG 업데이트 ② 버전 태깅 ③ 회귀 테스트 실행 ④ 릴리즈 노트 초안 순서를 캡슐화. 사용자가 "릴리즈 준비"라고 한 줄 던지면 마스터가 스킬에 따라 자동 실행한다.

### 14.5 메모리 활성화

서브에이전트 frontmatter에 `memory: project`를 추가하면 그 에이전트가 프로젝트 관행을 누적 학습한다. 처음에는 일반적인 패턴으로 일하지만, 사용이 누적되면서 프로젝트 고유의 코딩 스타일·테스트 관행·문서 톤을 익힌다.

다만 메모리는 컨텍스트를 늘리므로 비용을 만든다. 모든 에이전트에 메모리를 주는 것이 아니라, 도메인 학습이 가치 있는 에이전트(implementer, test-writer, doc-writer 등)에 선택적으로 적용한다.

### 14.6 진화의 리듬

처음 1~2주는 기본 8개로 충분하다. 3주차부터 사용 패턴을 보고 손본다. 한 달 정도 운영하면 어느 에이전트가 자주·드물게 쓰이는지, 어떤 도메인 분화가 필요한지 자연스럽게 드러난다.

진화는 점진적으로. 한 번에 여러 에이전트를 추가하거나 큰 구조 변경을 하지 말 것. 매 변경 후 라우팅이 잘 되는지, 비용이 예상대로인지 관찰하고 다음 변경으로 넘어간다.

### 프롬프트 시작

# 작업: 서브에이전트 + 스킬 오케스트레이션 시스템 초기 설정

## 1. 의도 (Intent)

이 프로젝트의 모든 향후 개발 작업을 비용 대비 품질 측면에서 최적화하기 위해, 마스터 세션과 분리된 서브에이전트 시스템을 구축한다. 마스터는 오케스트레이션(라우팅, 종합, 짧은 응답)만 담당하고 실제 작업은 작업 성격에 맞는 모델·역할의 서브에이전트에 위임한다. 마스터에서는 `/model` 교체를 절대 하지 않으며, 모델 다양성은 서브에이전트의 `model` 필드로만 표현한다.

## 2. 배경 원칙

- 마스터 세션 내에서 `/model` 교체 시 prompt cache가 모델별로 분리되어 무효화되며, 새 모델은 풀 입력을 다시 처리해야 한다. 따라서 마스터는 한 모델로 유지한다.
- 서브에이전트는 독립된 컨텍스트 윈도우에서 실행되어 마스터 컨텍스트를 오염시키지 않는다.
- 모델별 비용 비율(2026년 4월 기준): Haiku 4.5 ($1/$5) : Sonnet 4.6 ($3/$15) : Opus 4.7 ($5/$25). Opus 4.7은 새 토크나이저로 같은 텍스트가 1.0~1.35배 토큰을 더 사용한다.
- 결정 권한 분배 원칙: 비가역·고난도 → Opus / 일상 실행 → Sonnet / 대량·단순 → Haiku.

## 3. 완료 기준 (Acceptance Criteria)

다음 산출물이 모두 정확히 생성되어야 한다:

```
.claude/
├── agents/
│   ├── architect.md
│   ├── deep-debugger.md
│   ├── pr-reviewer.md
│   ├── implementer.md
│   ├── refactorer.md
│   ├── test-writer.md
│   ├── doc-writer.md
│   └── explorer.md
└── skills/
    └── sub-agent-routing/
        └── SKILL.md

CLAUDE.md (없으면 생성, 있으면 시스템 관련 섹션 추가)
```

추가 검증:
- `/agents` 명령으로 8개 에이전트가 모두 인식되는지 확인.
- explorer 에이전트로 검증용 호출 1회 수행 (예: 프로젝트 루트의 디렉토리 구조 요약 요청).

## 4. 제약 (Constraints)

- **기술 스택 종속 내용 절대 금지**: 특정 언어, 프레임워크, 빌드 도구, 테스트 라이브러리, 패키지 매니저를 가정하지 말 것. 모든 명세는 범용적이어야 한다.
- **추측 기반 작성 금지**: 프로젝트의 기존 구조나 관행은 explorer에 위임하여 먼저 확인할 것. 빈 프로젝트로 확인되면 바로 진행해도 된다.
- **이름 통일**: 각 에이전트의 `name` 필드와 파일명(확장자 제외)을 정확히 일치시킬 것.
- **`tools` 필드 명시 필수**: 비워두지 말 것. 보안·비용·실수 방지를 위해 각 에이전트가 필요한 도구만 허용한다.
- **`model` 필드는 반드시 명시**: `inherit`(기본값) 사용 금지. inherit는 마스터 모델을 그대로 상속하므로 비용 폭주의 주범이다.
- **`description` 필드는 자동 라우팅을 의도해 작성**: "Use PROACTIVELY for ..." 또는 "Use for ..." 패턴으로 시작하라. Claude는 description을 보고 위임 여부를 결정한다.
- **시스템 프롬프트(본문)에 출력 구조 명시 필수**: 마스터로 돌아오는 결과의 형식을 정의해두어 재질의 비용을 차단한다.
- **기존 `.claude/` 디렉토리가 있으면 충돌 점검**: 같은 이름의 에이전트가 이미 있다면 사용자에게 보고하고 진행 승인을 대기.

## 5. 서브에이전트 명세

각 항목은 frontmatter 핵심 필드와 본문에 포함해야 할 핵심 지침을 정의한다. 본문은 위 제약을 지키며 명세를 충실히 반영해 작성하라. description은 영어로 작성한다(라우팅 호환성을 위해).

### 5.1 architect — `model: opus`

```yaml
name: architect
description: Use PROACTIVELY for high-level design, architecture decisions, ADR drafting, API/schema design, and trade-off analysis on irreversible decisions. Read-only — does not modify code.
model: opus
tools: Read, Glob, Grep, WebSearch, WebFetch
```

본문 필수 요소:
- 역할: 비가역적 의사결정에 대한 깊은 추론과 구조화.
- 출력 구조 — ① 의사결정 요약 ② 옵션 비교 (최소 2개) ③ 선택 근거 ④ 위험·전제 ⑤ 후속 액션.
- 시작 시 "think carefully before responding"을 인지하고 충분히 사고할 것.
- 코드 수정 권한 없음(Edit/Write 도구 비포함). 분석·문서화 산출만 한다.

### 5.2 deep-debugger — `model: opus`

```yaml
name: deep-debugger
description: Use for race conditions, concurrency bugs, memory issues, mysterious heisenbugs, and cross-module bugs that require deep multi-step reasoning. NOT for simple stack-trace debugging — use implementer for those.
model: opus
tools: Read, Grep, Glob, Bash, Edit
```

본문 필수 요소:
- 역할: 단순 스택트레이스로 잡히지 않는 깊은 버그 진단·수정.
- 워크플로우 — 가설 수립 → 증거 수집 → 가설 검증 → 최소 수정.
- 출력 구조 — ① 근본 원인 ② 증거 ③ 수정 패치 ④ 회귀 테스트 제안.
- 가설이 3회 연속 실패하면 진행을 중단하고 보고할 것 (무한 추론 폭주 방지).

### 5.3 pr-reviewer — `model: opus`

```yaml
name: pr-reviewer
description: Use for security-sensitive, business-critical, or architecture-impacting code reviews. Focus on areas where missing a defect is costly. NOT for style/lint reviews — those belong to tooling.
model: opus
tools: Read, Grep, Glob, Bash
```

본문 필수 요소:
- 역할: 놓치면 비용이 큰 영역(보안, 동시성, 데이터 정합성, 인터페이스 계약, 권한)에 집중.
- 출력 구조 — ① 차단(blocker) 이슈 ② 강력 권고(should-fix) ③ 제안(nice-to-have) ④ 칭찬할 점.
- 스타일·포맷팅 지적 금지 (도구 영역).
- 코드 수정 금지 (읽기 전용).

### 5.4 implementer — `model: sonnet`

```yaml
name: implementer
description: Use PROACTIVELY for standard feature implementation when the spec is clear. The default workhorse for typical development tasks.
model: sonnet
tools: Read, Edit, Write, Glob, Grep, Bash
```

본문 필수 요소:
- 역할: 명세가 명확한 기능 구현.
- 입력 요구 — 의도, 제약, 완료 기준이 명세에 포함되어 있어야 한다. 모호하면 시도 전에 마스터에 질의할 것.
- 출력 구조 — ① 변경 파일 목록 ② 핵심 변경 요약 (3줄 이내) ③ 검증 방법.
- 프로젝트의 기존 코딩 관행을 explorer로 먼저 확인 후 따를 것.

### 5.5 refactorer — `model: sonnet`

```yaml
name: refactorer
description: Use for multi-file refactoring, code organization improvements, and structural changes that preserve behavior.
model: sonnet
tools: Read, Edit, Write, Glob, Grep, Bash
```

본문 필수 요소:
- 역할: 동작 보존(behavior-preserving) 리팩토링.
- 원칙 — 한 번에 한 가지 리팩토링 패턴, 매 단계 빌드·테스트 통과 확인.
- 광범위한 영향이 예상되는 구조 변경은 architect로 에스컬레이션할 것.
- 출력 구조 — ① 적용한 리팩토링 패턴 ② 변경 파일 목록 ③ 검증 결과.

### 5.6 test-writer — `model: sonnet`

- 출력 구조 — ① 추가된 테스트 파일 ② 커버하는 케이스 요약 ③ 발견한 엣지 케이스.

## 8. 작업 순서

1. **현황 파악** — explorer에 위임하여 프로젝트 루트 구조와 기존 `.claude/` 디렉토리 유무 파악.
2. **충돌 점검 보고** — 발견한 사항(특히 같은 이름의 기존 에이전트)을 1회 보고. 충돌이 있으면 진행 승인 대기. 빈 프로젝트면 바로 진행.
3. **파일 생성** — 위 명세에 따라 8개 에이전트 + 1개 스킬 + CLAUDE.md 생성/갱신.
4. **검증** — `/agents` 출력 확인 + explorer로 자기 자신 호출 테스트.
5. **요약 보고** — 생성·변경된 파일 목록과 사용자가 다음에 해야 할 일 1-3개.

## 부록 D. 용어집

**서브에이전트(sub-agent)**: 마스터 세션과 분리된 컨텍스트 윈도우에서 실행되는 특화 에이전트. `.claude/agents/<name>.md`로 정의.

**마스터 세션(master session)**: 사용자와 직접 대화하는 최상위 Claude Code 세션. 오케스트레이션 전담.

**스킬(SKILL)**: 마스터의 의사결정·작업 절차를 캡슐화한 단위. `.claude/skills/<name>/SKILL.md`로 정의. 자체 실행 단위가 아니라 참조되는 지식.

**effort**: Claude의 추론 깊이 조절 파라미터. low / medium / high / xhigh / max.

**xhigh**: Opus 4.7 전용 effort 레벨. high와 max 사이. Claude Code의 Opus 4.7 기본값.

**적응형 추론(adaptive thinking)**: 모델이 단계마다 추론 필요 여부를 자체 판단하는 모드. Opus 4.7의 유일한 thinking 모드.

**프롬프트 캐싱(prompt caching)**: 자주 반복되는 입력을 캐시해 재처리 비용을 줄이는 메커니즘. 캐시 히트는 표준 입력가의 10%. 모델별로 분리됨.

**ADR (Architecture Decision Record)**: 아키텍처 결정의 근거·옵션·결과를 기록한 문서. architect의 표준 산출물.

**CLAUDE.md**: 프로젝트 루트의 Claude Code 가이드 파일. 마스터 세션이 시작 시 자동 로드.

**inherit**: 서브에이전트의 `model` 필드 기본값. 마스터 모델을 상속. 비용 관점에서 위험한 기본값이므로 명시적 지정이 권장됨.

**위임 명세 4요소**: 의도 / 제약 / 완료 기준 / 관련 파일. 서브에이전트에 작업을 위임할 때 포함해야 할 항목.

**1M 컨텍스트**: 100만 토큰 컨텍스트 윈도우. Opus 4.7, Opus 4.6, Sonnet 4.6에서 표준 가격에 제공.

**Batch API**: 비동기 처리 API. 24시간 이내 응답 보장하며 모든 토큰에 50% 할인.

**Task budget (베타)**: Opus 4.7에서 도입된 에이전트 루프 전체의 토큰 상한 메커니즘. 폭주 방지용 안전장치.

**MCP (Model Context Protocol)**: 외부 도구·데이터 소스를 Claude에 연결하는 프로토콜. 서브에이전트별로 마운트 가능.
